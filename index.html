Using this app

This app helps testers and developers understand how different web components behave on mobile devices. It contains sample components with testing results. When a new OS is released, the data will be updated.

Some of the components are created, others were used or modified.

The best way to use the app is experimenting with component behavior on different mobile devices, or modifying the code and see how the behavior changes.

Note: in your CSS file, set: text-transform:none;

    When jQuery is loaded, buttons have two touchpoints. The first is the label of the button itself that cannot be activated. The second is the actual button, with the same label. This one functions properly. This solution is usable, but does not fully meet accessibility requirements.

    There is a solution where the label only shows up once, but it requires using a link with a button ARIA role. This only meets accessibility requirements if the link is also designed as a button. This can be achieved by using the jQuery ui-btn class for the link.

    Open Panel

    Panels are generally accessible, however, it is important to note that the data-rel should be set to close. This will add a button to the panel that allows keyboard users to close the panel, otherwise it will only work using a swipe.

    The panel can be placed anywhere in the page code, but depending on the location, it will show at different places. When placed above the page header, the opened panel will show in the DOM before the header. When placed below the header, before the main content, the panel will render at the end of the page. See the panel below header page.

    It is important to provide verbal indication to the user that a panel is opening, as it may not be obvious where to look for the newly displayed information.

    The button used to open the panel, and the close button show as links. Therefore, it is more appropriate to add an ARIA button role to these links, for an example, see the panel below header page.
